Zombie Game
Unity C#

I created this project for my final junior design assignment. I always thought it would be interesting to create a game and I wanted to see how well I would do. For the time that I had, it ended up being more ambitious then I realized. However, I enjoyed learning how games are made and the design practices that come with it.

The game starts with you in the center of a small map. Zombies spawn during each round until they hit a spawn cap. Once they do, you must kill the current amount for more to spawn. Once you've killed all the zombies of that round, the next round will start with stronger zombies. Throughout the map there are other guns and perks you can buy to help you survive. By purchasing doors, an interactable, you can access areas to buy stronger weapons and perks to benefit you. In the bottom left of the HUD you will see a red bar for health and a blue bar for stamina. In the bottom middle, are the rounds. At the bottom right in green is your money, in yellow is the ammo currently in your gun, and in white is reserve ammo.

Upon death you will see a death screen with how many zombies you killed, and what round you made it to.

Brief Overview of Each Script

  • Spawner: Spawns zombies at certain spawn points. Will only spawn as many as are allowed to be pooled. After all zombies are killed in a round, it spawns new zombies with more health and changes the round.
  • Object Pool: Creates a list of objects at start time and sets them to false. If an object is able to be pooled it is called from this function.
  • Move Settings: Standard settings for movement.
  • IDamageable: Interface that makes the player and zombies damageable that is implemented in their respective classes.
  • Guns: Creates bullets at an attack point. Sets bullet damage. Moves camera based on recoil. Will only shoot if there are bullets in the magazine. Reloads if there is more ammo. Creates sounds on reload and shoot.
  • Custom Bullet: Keeps track of variables for the bullet. Destroys on collision with enemy and does damage to them.
  • Crosshair: Changes the size of the crosshair based on character movement and aiming down sights.
  • Attack Radius: Damages damageable game objects that are within the attack radius after a certain amount of time.
  • Player Move: Adds force to the player character controller based on key inputs. Regenerates and drains stamina based on running and jumping.
  • Player: Implements IDamageable. Takes damage and sets the game object to false on death (health goes to zero). Regenerates health. Shows player health bar. Shows game over screen with final round, total enemies killed and game over.
  • CamMove: Moves the camera up and down and rotates the player character controller left and right based on mouse input.
  • IInteractable: Interactable interface. Creates cost, prompt when interacting with an interactable, and function for interacting.
  • WeaponBuy: Implements IInteractable. Replaces values of gun script with new ones for specified guns. Destroys the current gun game object and creates a new one as the child of the camera. If already purchased, refills ammo for the gun.
  • Perks: Implements IInteractable. Changes a value within the gun script.
  • Door: Implements IInteractable. Sets game object to false.
  • Interactor: Checks if the interactor sphere overlaps with the interactable sphere and adds it to a interactable list if so. If the player has enough points and interacts with the interactable, it is called IInteractable class.
  • Enemy Movement: Makes the enemy follow the target. Revives zombie when respawned.
  • Enemy: Implements IDamageable. Sets the enemy attack radius. Rotates the zombie to look at the player. Takes damage and if the zombie health is below zero will despawn it and add kill count to the spawner.

If I was able to go back and do it again, there are a few things I'd do differently and/or add. First I'd make the gun class an interface so I could more easily make unique weapons. I wouldn't use Unity terrain because it's difficult to work with, doesn't look good, and is a little glitchy in game. I would change any public variables to serialized fields simply because that is the “right” way to do it. I would change any gameObjects that are instantiated, created during runtime, to poolable objects, objects that are created pre runtime and set to inactive, in order to increase performance of the game and implement real software practices. And if I had more time I would create a start menu, pause menu, get better sounds, animations and objects so the game doesn't feel like a hodge podge of random free assets.
